# Michael Shen's Personal Blog

This is a static website hosted on GitHub Pages at [imichaelen.github.io](https://imichaelen.github.io).

## Site Structure

This site consists of two main parts:

### 1. Main Blog (`/`)

A personal blog with posts, archives, tags, and an about page. The site is fully static HTML/CSS/JS.

**Key directories:**

- `/post/` - Individual blog post pages
- `/archives/` - Archive listing page
- `/tags/` - Tag listing page
- `/images/` - Avatar and site images
- `/post-images/` - Images used in blog posts
- `/styles/` - CSS stylesheets
- `index.html` - Homepage
- `atom.xml` - RSS feed

### 2. Demo Site (`/projects/demo/`)

An autonomous research workflow demonstration showcasing AI-driven first-principles research. This is a self-contained static site with interactive visualizations.

**Demo features:**

- Interactive idea tree visualization
- Workflow timeline
- FPilot execution logs
- All data is sanitized and static (no external API calls)

## How to Update Content

### Updating the Main Blog

Since this is a static site (no build process), you edit HTML files directly:

#### Add a New Blog Post

1. Create a new directory in `/post/` with your post slug (e.g., `/post/my-new-post/`)
2. Copy an existing `index.html` from another post as a template
3. Edit the HTML to update:
   - Page title (`<title>`)
   - Meta description and keywords
   - Post title in the `<h2 class="post-title">` section
   - Post content
   - Post date
   - Tags (if any)
4. Update `/index.html` to add your post to the homepage listing
5. Update `/archives/index.html` to add your post to the archives
6. Update `/atom.xml` to add your post to the RSS feed
7. If you added new tags, create or update tag pages in `/tag/` and `/tags/index.html`

#### Edit an Existing Post

Navigate to `/post/<post-name>/index.html` and edit the HTML directly.

#### Update Site Metadata

- **Site title/description**: Edit the `<div class="site-header">` section in all HTML files
- **Navigation menu**: Edit the `<div class="menu-container">` section in all HTML files
- **Avatar**: Replace `/images/avatar.png`
- **Social links**: Edit the `<div class="social-container">` sections

### Updating the Demo Site

The demo site (`/projects/demo/`) has its own documentation in `/projects/demo/README.md`. Key points:

1. **Update demo data**: Run `python scripts/generate_demo_data.py` (if the script exists) to regenerate sanitized data
2. **Edit demo content**: Modify files in `/projects/demo/` directly
3. **Test locally**: Open `/projects/demo/index.html` in a browser or run a local server

## How to Preview Locally

### Quick Preview (Static Files)

Simply open `index.html` in your web browser. Note: some features may not work correctly due to CORS restrictions.

### Local Server (Recommended)

Run a simple HTTP server to test everything properly:

**Python 3:**

```bash
python3 -m http.server 8000
```

**Python 2:**

```bash
python -m SimpleHTTPServer 8000
```

**Node.js (if you have `npx`):**

```bash
npx http-server -p 8000
```

Then visit:

- Main site: `http://localhost:8000/`
- Demo site: `http://localhost:8000/projects/demo/`

## How Deployment Works

This site is automatically deployed via **GitHub Pages** using the following configuration:

- **Repository**: `imichaelen/imichaelen.github.io`
- **Branch**: `master`
- **Source**: Root directory (`/`)
- **URL**: `https://imichaelen.github.io`

### Deployment Process

1. Make your changes locally (edit HTML files)
2. Test using a local server (see above)
3. Commit your changes:
   ```bash
   git add .
   git commit -m "Update: describe your changes"
   ```
4. Push to GitHub:
   ```bash
   git push origin master
   ```
5. GitHub Pages automatically serves the updated content (usually within 1-2 minutes)

### Verify GitHub Pages Settings

To confirm your GitHub Pages configuration:

1. Go to `https://github.com/imichaelen/imichaelen.github.io/settings/pages`
2. Ensure:
   - **Source**: Deploy from a branch
   - **Branch**: `master` / `(root)`
   - **Custom domain**: (leave blank unless you have a custom domain)

## Tips for Managing Static HTML Sites

### Pros

✅ Simple deployment (no build step)  
✅ Fast loading (pre-rendered HTML)  
✅ Easy to understand and debug  
✅ Works everywhere (no dependencies)

### Cons

❌ Repetitive editing (headers/footers in every file)  
❌ No Markdown support (must write HTML)  
❌ Manual index updates when adding posts

### Future Migration Options

If you want to reduce repetition while keeping things simple, consider:

1. **Jekyll** (GitHub Pages built-in)
   - Write posts in Markdown
   - Templates for headers/footers
   - Automatic archive/tag generation
   - No external build step needed (GitHub builds automatically)

2. **11ty (Eleventy)** (External build)

   - Minimal static site generator
   - Very flexible, plain JavaScript
   - Requires local build step before pushing

3. **Custom scripts**
   - Write a simple script to generate HTML from Markdown
   - Keep full control, minimal dependencies

## Daily Newspaper (Automated)

This repo includes a small, deterministic pipeline that generates a daily arXiv digest and commits it back to `master` so GitHub Pages updates automatically.

**Where it outputs**

- `daily/index.md` (Latest + Archive)
- `daily/issues/YYYY-MM-DD.md` (JST date)
- `data/state.json` (committed; tracks seen IDs + last run)

Intermediate files like `data/collected.json` and `data/summaries.json` are intentionally gitignored.

### Configure what you track

Edit `config.yaml`:
- `arxiv.queries`: add/remove categories/keywords (Lucene query syntax used by arXiv API)
- `issue.lookback_days`: include only papers published within the last N days (set `0` to disable)
- `issue.featured_papers`: how many papers get full summaries on the page
- `llm.trend_enabled`: generate a “Trend” section (requires API key)
- `llm.digest_enabled`: generate a “What’s New Today” newspaper-style digest (requires API key)

### How to run locally

```bash
python3 -m venv .venv
source .venv/bin/activate
python -m pip install -r requirements.txt

python scripts/collect.py --config config.yaml
python scripts/summarize.py --config config.yaml   # optional; requires OPENAI_API_KEY or DEEPSEEK_API_KEY
python scripts/render.py --config config.yaml
```

Notes:
- Export `OPENAI_API_KEY` and/or `DEEPSEEK_API_KEY` in your shell environment (you can source your local secrets file yourself; do not commit secrets).
- Local `python -m http.server` will show the `.md` files as text; GitHub Pages renders them as HTML.

### GitHub Actions

Workflow: `.github/workflows/daily.yml` runs daily at **07:00 JST** (`22:00 UTC`) and uses repo secrets:
- `DEEPSEEK_API_KEY` (preferred)
- `OPENAI_API_KEY`
